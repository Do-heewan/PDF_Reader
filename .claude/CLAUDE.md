# CLAUDE.md

> **어떤 개발 서적이든**(영문·한글 불문) PDF를 챕터별로 분할하고 한국어 학습노트로
> 정리해 주는 **범용** 로컬 도구.
> 이 문서는 Claude Code가 이 프로젝트를 **개발하고 운영**할 때 따르는 단일 기준이다.
> 빈 폴더에서 시작하더라도 이 문서만으로 전체 구현이 가능하도록 작성되었다.

---

## 1. 프로젝트 목표

- **무엇을**: `data/` 에 넣은 **임의의 개발 서적 PDF**(영문이든 한글이든)를 챕터별로
  분할하고, 챕터마다 **한국어 학습노트(`.md`)**를 생성한다.
- **왜**: 책 한 권을 챕터 단위로 꾸준히 학습하고 그 내용을 자기 말로 재구성해
  블로그·노트에 정리하기 위해서다. 특정 책·주제에 묶이지 않은 **범용 학습노트 생성기**가
  목표이며, 책이 바뀌어도 코드·커맨드를 고치지 않고 그대로 재사용할 수 있어야 한다.
- **설계 불변식**: 어떤 도구도 특정 책(쿠버네티스 등)·특정 기술 스택을 전제하지 않는다.
  책 제목·챕터 수·전문 용어는 모두 **입력 PDF에서 동적으로** 끌어온다.
- **현재 범위**: 로컬 프로젝트. 블로그 자동 게시는 추후 과제(지금은 `.md` 생성까지).

> **참고 — 개발자(저장소 소유자)의 개인 사용례**: 쿠버네티스 원서(약 350쪽, 30챕터)를
> 하루 1챕터씩 학습하는 데 쓰고 있다. 이는 **하나의 사용 예시일 뿐 프로젝트의 목적이
> 아니다.** 문서·코드·예시는 쿠버네티스에 특화되지 않도록 일반적으로 작성한다.

## 2. 핵심 설계 원칙 — 2단계 분리

이 프로젝트의 모든 구조는 아래 분리에서 나온다. **두 역할을 섞지 말 것.**

1. **분할 (코드 / AI 불필요)** — `scripts/extract_chapters.py` 가 PDF 목차를 읽어
   챕터 경계(페이지 범위)를 찾고 챕터별 PDF로 자른다. 결정론적이라 결과가 항상 같고,
   수백 쪽짜리 책 전체를 매번 Claude에 넣지 않아도 된다.
2. **정리 (AI)** — 잘린 챕터 PDF **하나만** Claude Code(`/study`)가 읽고 학습노트를 만든다.

→ 토큰·비용 절약 / 그림 보존(텍스트만 뽑으면 다이어그램이 소실됨) / 재현성 확보.

## 3. 폴더 구조

```
.
├── data/                       # 학습할 책 PDF 1개 (입력) — 책 무관, 파일명 자유
│   └── <book>.pdf
├── scripts/
│   └── extract_chapters.py     # 챕터 분할기 (구성요소 ①)
├── chapters/                   # 분할 결과 (자동 생성)
│   └── chapterNN/
│       ├── source.pdf          # 분할 원본 — 읽기 입력, 수정 금지
│       ├── source.txt          # 텍스트 추출본 (--text 옵션일 때만)
│       └── chapterNN.md        # 학습노트 (/study 산출물)
├── .claude/
│   ├── CLAUDE.md               # 이 문서
│   └── commands/
│       └── study.md            # /study 슬래시 커맨드 (구성요소 ②)
├── toc.json                    # 챕터↔페이지 매핑 (자동 생성·수동 교정 가능)
└── requirements.txt
```

> 챕터 폴더를 루트에 평탄하게 두려면 `extract_chapters.py` 의
> `DEFAULT_OUT = ROOT / "chapters"` 를 `DEFAULT_OUT = ROOT` 로 바꾼다. 다만 챕터 폴더가
> `data/`·`scripts/` 와 섞이므로 `chapters/` 로 묶는 쪽을 권장한다.

## 4. 기술 스택 / 실행

- Python 3.10+
- `pymupdf` — PDF 목차 읽기 + 분할 + 텍스트 추출을 모두 담당(단일 의존성).

```bash
pip install -r requirements.txt                 # requirements.txt 내용: pymupdf>=1.24
python scripts/extract_chapters.py --inspect    # 챕터 감지 결과만 확인(파일 생성 안 함)
python scripts/extract_chapters.py --text       # 분할 실행(+텍스트 추출)
```

## 5. 구성요소 ① — 챕터 분할기 `scripts/extract_chapters.py`

**책임**: PDF → 챕터별 `source.pdf` + `toc.json`.
요약·정리는 하지 않는다(그건 `/study` 의 몫).

### CLI 인터페이스

| 플래그 | 동작 |
|---|---|
| (없음) | 전체 챕터 분할 |
| `--inspect` | 감지된 챕터 구조만 출력, 파일 생성 안 함 |
| `--chapter N` | N번 챕터만 (재)분할 |
| `--text` | `source.pdf` 와 함께 `source.txt` 도 추출 |
| `--reindex` | `toc.json` 무시하고 목차 재감지 |
| `--pdf PATH` | PDF 경로 직접 지정(생략 시 `data/` 자동 탐색) |
| `--out-dir DIR` | 챕터 폴더 생성 위치(기본 `chapters/`) |

### 챕터 감지 — 우선순위

1. `toc.json` 이 있으면 그대로 사용(사람이 교정한 정답). `--reindex` 면 무시.
2. **PDF 내장 목차(outline)**: 제목이 `Chapter N` / `제N장` / `N장` 패턴과 맞는 항목을
   우선 채택(3개 이상일 때). 안 맞으면 가장 얕은 레벨 항목을 챕터로 간주.
3. 목차가 없으면 각 페이지 상단(앞 6줄)에서 위 패턴을 스캔.
4. 그래도 못 찾으면 종료하며 `toc.json` 수동 작성을 안내.

### 페이지 범위 규칙

챕터 k = `[start_k, start_{k+1} - 1]`, 마지막 챕터는 문서 끝까지.
목차 페이지 번호는 **1-based 물리 페이지**로 다룬다.

### 출력물

- `chapters/chapterNN/source.pdf` — PyMuPDF `insert_pdf(from_page, to_page)` 로 분할.
- `chapters/chapterNN/source.txt` — `--text` 일 때만.
- `toc.json` — `{ "source": str, "chapters": [{ "index", "title", "start", "end" }] }`.

### 구현 주의

- import 는 `import fitz` 우선, 실패 시 `import pymupdf as fitz` 로 폴백.
- `data/` 에 PDF가 0개거나 2개 이상이면 명확한 메시지로 종료.
- 헤더·푸터에 챕터 번호가 반복되는 PDF에서 같은 챕터 번호가 중복 감지되지 않도록 처리.

## 6. 구성요소 ② — `/study` 커맨드 `.claude/commands/study.md`

`/study NN` 의 동작:

1. 번호를 두 자리로 정규화(`3` → `03`). 대상 폴더는 `chapters/chapterNN/`.
2. `chapters/chapterNN/source.pdf` 를 읽는다 — 텍스트뿐 아니라 **그림·다이어그램까지**
   해석한다. `source.txt` 가 있으면 보조로 참고.
3. 아래 §7 규칙·템플릿대로 `chapters/chapterNN/chapterNN.md` 를 작성.
4. 마지막에 **셀프 체크 3문항만** 사용자에게 보여주고 스스로 답해보도록 안내.

**원칙: 한 번에 한 챕터만.** 여러 챕터를 몰아서 생성하지 않는다(매일 학습이 목적).
`source.pdf` 가 없으면 먼저 `python scripts/extract_chapters.py` 를 안내한다.

## 7. 학습노트 작성 규칙

`source.pdf` 는 그림·다이어그램까지 포함된 챕터 원본이다. 텍스트만 보지 말 것.
**원문이 영어든 한국어든** 결과 노트는 항상 한국어로 작성한다.

- 책 문장을 옮겨 적지 않는다. 핵심을 **자기 말로 재구성**(저작권·학습 양면에서 중요).
- **개념 설명서(서술형 글)로 쓴다 — 가장 중요한 규칙.** "정의 한 줄 / 왜 필요한지 한 줄"
  식의 골격 나열을 만들지 않는다. 이어지는 문단으로, 개념이 무엇이고 어떤 문제 때문에
  등장했으며 어떻게 동작하고 다른 개념과 어떻게 맞물리는지를 책 내용을 바탕으로 **내가
  이해한 바를 남에게 설명하듯** 풀어 쓴다(비유·예시 활용). 불릿은 진짜 목록(특성 나열 등)일
  때만 보조로 쓰고, 설명의 본체는 문단이어야 한다.
- 코드·명령어·설정·매니페스트는 코드 블록으로. 식별자(클래스·함수·명령·리소스 이름,
  예: `Pod`, `kubectl apply`, `useEffect`, `SELECT`)는 백틱으로 감싼다. 책의 주제에 맞는
  언어·도구를 그대로 쓴다(특정 스택을 가정하지 않는다).
- 군더더기·홍보성 표현 없는 한국어 기술 문체. 단 골격만 남기지 말고 흐름이 이어지는 설명문으로.
- 그림은 '무엇을 보여주는 그림인지 + 흐름'을 글/표/화살표로 재현하고 의미를 해석한다.
- 그 챕터를 처음 보는 사람도 이 글만 읽고 개념을 이해할 정도의 분량·친절함. 다만 불필요하게
  늘이지 않는다.

### 학습노트(개념 설명서) 템플릿 (`chapterNN.md`)

각 절의 안내문은 **무엇을 쓸지에 대한 지시**다. 실제 노트에서는 지우고 서술형 문단으로 채운다.

````markdown
# {NN}. {챕터 제목}

> 학습일: {YYYY-MM-DD} · 원문 pp. {start}–{end}

## 이 챕터가 답하는 질문
챕터가 풀려는 문제·질문을 한두 문장으로 던져, "왜 이걸 배우지?"에 먼저 답한다.

## 큰 그림
챕터 전체를 관통하는 줄거리를 한 문단으로. 어떤 개념이 어떤 순서로 이어져 무엇을
향하는지 지도처럼 먼저 보여준다. 선행 지식·직전 챕터 맥락도 여기서 자연스럽게 짚는다.

## {개념 1 제목 — 한국어(원어)}
설명문으로 풀어 쓴다. 한 줄 정의로 끝내지 말고: 무엇인지 + 어떤 문제로 등장했는지 +
어떻게 동작하는지 + 다른 개념과 어떻게 맞물리는지를, 비유·예시를 곁들여 문단으로 엮는다.

## {개념 2 제목}
(같은 방식. 개념 수만큼 절을 늘린다.)

## 개념들이 맞물리는 방식
따로 설명한 개념들이 어떻게 연결되어 하나의 그림을 이루는지 한 문단으로. (단일 개념이면 생략.)

## 코드 · 명령어 · 설정으로 보기
```text
(챕터 주제에 맞는 코드·명령·설정 예시. 책의 언어·도구를 그대로 사용)
```
예시가 무엇을 하는지, 왜 그렇게 쓰는지를 코드 블록 아래 문장으로 설명한다.

## 다이어그램으로 이해하기
핵심 그림의 구조·흐름을 글/표/화살표로 재구성하고, 그림이 말하려는 바를 해석한다.

## 흔히 헷갈리는 점 · 함정
혼동하기 쉬운 구분·함정을, 왜 헷갈리는지까지 짚어 서술형으로 설명한다.

## 셀프 체크 (보지 않고 답해보기)
1.
2.
3.

## 한 줄 회고 — 직접 채우기
> (학습 후 본인이 작성: 가장 인상 깊었던 점 / 실무에서 쓸 곳)
````

'셀프 체크'는 **질문만 남기고 답은 비운다.** '회고'는 사용자가 직접 채운다.
노트가 정답을 다 떠먹여주면 블로그에 옮겨도 머리에 남지 않는다 — 이 두 섹션이
수동 학습을 능동 학습으로 바꾸는 장치다.

## 8. 진행 관리

- `toc.json`: 감지가 틀리면 `start`/`end` 를 직접 고친 뒤 재분할. 목차가 아예 없는
  PDF라면 이 파일을 처음부터 직접 작성해도 된다. `source` 에는 실제 입력 PDF 파일명이 들어간다.

```json
{
  "source": "<book>.pdf",
  "chapters": [
    { "index": 1, "title": "Chapter 1. ...", "start": 1,  "end": 12 },
    { "index": 2, "title": "Chapter 2. ...", "start": 13, "end": 28 }
  ]
}
```

## 9. 개발 작업 지침 (Claude Code)

- **현재 상태**: 구성요소 ①②는 모두 구현 완료. `data/` 에 책 1권이 들어 있고
  `chapters/chapter01`–`chapter30` 으로 이미 분할되어 있다(`toc.json` 존재).
  즉 빈 폴더가 아니라 **운영 중인 저장소**다.
- **실행 환경**: Python 3.10+. Windows/PowerShell 기준이며 `python` 대신 `py` 런처를
  써도 된다. `extract_chapters.py` 는 cp949 콘솔에서 한글이 깨지지 않도록 stdout/stderr 를
  의도적으로 UTF-8 로 재설정한다([scripts/extract_chapters.py:24-29](scripts/extract_chapters.py#L24-L29)) —
  이 코드를 제거하지 말 것.
- **검증 방법**: 자동화된 테스트는 없다. 분할 로직은 `data/` 의 실제 책으로 확인한다 —
  `python scripts/extract_chapters.py --inspect` 의 감지된 챕터 수·페이지 범위가
  `toc.json` 과 맞는지, `--chapter N` 재분할 후 `source.pdf` 페이지 수가
  `end - start + 1` 과 같은지 비교한다. (작은 테스트 PDF가 필요하면 PyMuPDF의
  `doc.set_toc([...])` 로 목차 있는 더미 PDF를 만들어 같은 검증을 할 수 있다.)
- **확장 포인트(추후)**: 다른 책으로 재사용(이미 책-무관 설계), 블로그 자동 게시 단계
  추가, 다이어그램 이미지 별도 추출 후 노트에 삽입.
- **금지**: 비밀번호·토큰 등 자격증명을 코드·설정 파일에 넣지 않는다.
