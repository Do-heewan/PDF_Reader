# /study — 챕터별 학습노트 생성

너는 사용자가 **어떤 개발 서적이든**(영문·한글 불문) 한 챕터씩 학습하도록 돕는다.
대상 책은 `data/` 에 들어 있는 PDF이며, 결과 노트는 **항상 한국어**로 작성한다.
특정 책·기술 스택을 가정하지 말고, 책 제목·용어·코드 언어는 그 챕터 PDF에서 그대로 끌어온다.
이번 호출에서 정리할 챕터 번호는 `$ARGUMENTS` 다. (비어 있으면 어느 챕터인지 사용자에게 묻고 멈춘다.)

## 절차

1. **챕터 번호 정규화**: `$ARGUMENTS` 의 숫자를 두 자리로 만든다 (`3` → `03`, `12` → `12`).
   대상 폴더는 `chapters/chapter{NN}/`, 노트 파일은 `chapters/chapter{NN}/chapter{NN}.md`.

2. **원본 확인**: `chapters/chapter{NN}/source.pdf` 가 있는지 본다.
   - 없으면 노트를 만들지 말고, 다음을 안내하고 멈춘다:
     ```bash
     python scripts/extract_chapters.py            # 아직 분할 안 했을 때
     python scripts/extract_chapters.py --chapter {N}   # 해당 챕터만 다시 분할
     ```

3. **읽기**: `chapters/chapter{NN}/source.pdf` 를 Read 도구로 읽는다.
   - **텍스트뿐 아니라 그림·다이어그램·표까지** 해석한다. 아키텍처 그림은 학습의 핵심이다.
   - `chapters/chapter{NN}/source.txt` 가 있으면 텍스트 보조 참고용으로만 함께 본다
     (그림 정보가 없으므로 PDF가 항상 우선).

4. **노트 작성**: 아래 §규칙·§템플릿대로 `chapters/chapter{NN}/chapter{NN}.md` 를 Write 한다.
   - `pp.` 범위와 챕터 제목은 `toc.json` 에서 가져온다 (없으면 PDF 내용으로 추정).
   - 학습일은 오늘 날짜(`YYYY-MM-DD`).

5. **마무리**

## 원칙

- **한 번에 한 챕터만.** 여러 챕터를 몰아서 생성하지 않는다. 한 챕터씩 꾸준한 학습이 목적이다.
- 책 문장을 그대로 옮기지 않는다. 핵심을 **자기 말로 재구성**한다 (저작권·학습 양면).
- **원문이 영어든 한국어든 노트는 한국어로** 쓴다.
- **개념 설명서(서술형 글) 형태로 쓴다.** 가장 중요한 규칙이다. "정의: …, 왜 필요한가: …"
  식의 한 줄짜리 골격 나열을 만들지 않는다. 대신 **이어지는 문단**으로, 이 개념이 무엇이고
  어떤 문제 때문에 등장했으며 어떻게 동작하고 다른 개념과 어떻게 맞물리는지를, 책 내용을
  바탕으로 **내가 이해한 바를 남에게 설명하듯** 풀어 쓴다. 필요하면 비유·예시를 곁들인다.
  불릿은 진짜 목록(특성 나열 등)일 때만 보조로 쓰고, 설명의 본체는 문단이어야 한다.
- 코드·명령어·설정·매니페스트는 코드 블록으로. 식별자(클래스·함수·명령·리소스 이름)는
  백틱으로 감싼다. 책의 주제에 맞는 언어·도구를 그대로 쓰고 특정 스택을 가정하지 않는다.
- 군더더기·홍보성 표현 없는 한국어 기술 문체. 다만 골격만 남기지 말고 **흐름이 이어지는
  설명문**으로 쓴다.
- 그 챕터를 처음 보는 사람도 이 글만 읽고 개념을 이해할 수 있을 정도의 분량과 친절함.
  단, 불필요하게 늘이지 않는다.

## 학습노트(개념 설명서) 템플릿 (`chapter{NN}.md`)

각 절의 안내문은 **무엇을 쓸지에 대한 지시**다. 실제 노트에서는 안내문을 지우고
서술형 문단으로 채운다.

````markdown
# {NN}. {챕터 제목}

> 학습일: {YYYY-MM-DD} · 원문 pp. {start}–{end}

## 지난 시간에 학습한 내용
NN-1 번째 학습 노트인 `chapterNN/chapter{NN-1}.md` 파일을 읽어 간단한 요약을 한다. 이번 챕터와 어떤 개념이 맞물리는지 체크.
없으면 패스한다.

## 이번 챕터에서 배우게 될 내용
이번 챕터가 풀려는 문제나 답하려는 질문을 한두 문장으로 던진다. 주로 어떤 내용을 다룰지 독자들에게 미리 암시한다.

## {개념 1 제목 — 한국어(원어)}
이 개념을 **설명문**으로 풀어 쓴다. 한 줄 정의로 끝내지 말고, 다음이 자연스럽게
이어지는 문단(들)로 엮는다.
- 이게 무엇인지, 그리고 **어떤 문제 때문에 등장했는지**(없으면 무엇이 곤란해지는지).
- 어떻게 동작하는지, 핵심 메커니즘을 단계나 흐름으로.
- 앞의 개념이나 다른 도구와 **어떻게 맞물리는지**, 적절한 비유나 예시.
- 코드 관련 설명이 있다면, 해당 코드 전체를 작성 후 아래에 설명을 작성한다.

남에게 말로 설명하듯, 읽으면 머릿속에 그림이 그려지도록 쓴다.

## {개념 2 제목}
(같은 방식으로 서술. 개념 수만큼 절을 늘린다.)

## 큰 그림(또는 결론)
챕터 전체를 관통하는 줄거리를 한 문단으로 서술한다. 
챕터 마지막 개념에 **Discussion** 파트가 있을 것이다. Discussion의 내용을 참고해서 작성하면 좋다(Kubernetes Patterns 한정). 없다면 넘어가도 좋다.
어떤 개념들이 어떤 순서로 이어지며 결국 무엇을 향하는지를, 마지막 개념 요약 하듯이 지도처럼 보여준다.
````
